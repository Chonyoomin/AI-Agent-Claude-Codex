# Agentic AI Coding Loop

Agentic AI Coding Loop is a local orchestration project for running a controlled, phase-gated build-review-fix workflow between Codex and Claude Code.

Codex acts as the planner, task-state manager, reviewer, validator, and fix-prompt generator. Claude Code acts as the implementation agent for the active phase only. A local orchestrator manages prompts, artifacts, validation logs, and loop state. A human approves movement between phases and separately approves any eventual commit.

## Goal

Reduce manual copy/paste between Codex and Claude Code while keeping the workflow auditable, scoped, and safe.

## MVP

The first version focuses on a small local loop:

- read a human-provided objective
- let Codex maintain `TASK.md` and active phase state
- generate a focused Claude implementation prompt
- let Claude implement the requested phase
- capture Claude's structured summary
- capture `git diff` and `git status`
- run validation commands and save logs
- let Codex review the prompt, summary, diff, and logs
- generate a fix prompt if needed
- stop after each approved phase until the human starts the next one

## Workflow

This section describes the end-to-end workflow the shipped orchestrator at `scripts/agent_loop.py` (Phase 3A-3E) drives on a clean clone. New operators should start with [docs/usage.md](docs/usage.md) for the clean-clone walkthrough; the `## Running The Loop Manually (Phase 1)` section below is preserved as a hand-driven fallback against the same artifact formats.

1. Human provides the project objective or updates the desired outcome.
2. Codex updates `TASK.md`, `.agent-loop/current-task.md`, and `.agent-loop/current-phase.md` for the active phase.
3. Codex writes `.agent-loop/claude-prompt.md`.
4. Claude Code implements only the active phase and writes `.agent-loop/claude-summary.md`.
5. The orchestrator captures diff, status, and validation logs.
6. Codex writes `.agent-loop/codex-review.md` with one verdict:
   - `APPROVED_FOR_HUMAN_REVIEW`
   - `NEEDS_FIXES`
   - `FAILED_REQUIRES_HUMAN`
7. If fixes are required, Codex classifies review findings by ownership: supported Codex-owned follow-up is handled directly, and Claude-owned fixes are synchronized into `.agent-loop/fix-prompt.md`.
8. Claude applies targeted fixes within the same phase.
9. The loop repeats until the phase is approved, escalated, or reaches max cycles.
10. When a phase is approved, the system stops until the human starts the next phase.
11. The system never commits without separate human approval.

## Safety

- no auto-commit
- no auto-push
- no hidden validation failures
- no trusting summary text over actual diffs and logs
- no phase advancement without human approval
- no final commit without human approval

## Ownership Rules

- Claude Code is the default implementation editor for code, tests, scripts, configuration, and other non-state repository files during an active phase
- Codex should normally route implementation changes to Claude Code and edit directly only when the issue has been intentionally assigned to Codex
- planning-state hygiene issues should be treated as Codex-resolved and fixed directly by Codex
- `.agent-loop/loop-state.json`, `.agent-loop/orchestrator.log`, and the evidence files are owned by the orchestrator or the scripts that generate them; they must not be fabricated by hand outside explicitly assigned implementation work
- if ownership is ambiguous, the change should be routed to Claude Code or escalated for human clarification rather than guessed
- when continuing an active implementation phase, Codex should first summarize the remaining implementation gap, classify it as Claude-resolved or Codex-resolved, and if it is Claude-resolved write the precise Claude Code task prompt before implementation proceeds
- when the human starts a new phase or sub-phase, Codex should first provide a rundown of what will be implemented, split the work into Codex-resolved and Claude-resolved responsibilities, send the Claude-resolved portion as a precise Claude Code task prompt, and then perform only the Codex-resolved portion directly

## Running The Loop Manually (Phase 1)

Phase 1 documented the manual file-based loop. The by-hand workflow below is preserved as a fallback even though the shipped orchestrator (Phase 3A-3E) is now the canonical driver and is documented in [docs/usage.md](docs/usage.md). Run the manual loop against the artifact formats defined in `AGENTS.md` when the orchestrator cannot be invoked (hand-driven cycle reproduction, audit walk-throughs). One full manual cycle:

1. Codex updates `TASK.md`, `.agent-loop/current-task.md`, and `.agent-loop/current-phase.md` for the active phase.
2. Codex writes the implementation prompt to `.agent-loop/claude-prompt.md` using the required Claude task format.
3. A human pastes that prompt into Claude Code.
4. Claude Code implements only the active phase and writes `.agent-loop/claude-summary.md` using the required Claude implementation summary format.
5. A human captures evidence into `.agent-loop/`:
   - `git diff > .agent-loop/git-diff.patch`
   - `git status > .agent-loop/git-status.log`
   - test, lint, typecheck, and build output to `.agent-loop/test-output.log`, `.agent-loop/lint-output.log`, `.agent-loop/typecheck-output.log`, `.agent-loop/build-output.log` (or record "Not run" explicitly).
6. A human pastes the prompt, summary, diff, and logs into Codex.
7. Codex writes `.agent-loop/codex-review.md` with exactly one verdict: `APPROVED_FOR_HUMAN_REVIEW`, `NEEDS_FIXES`, or `FAILED_REQUIRES_HUMAN`.
8. If the verdict is `NEEDS_FIXES`, Codex-owned follow-up is applied first where supported, `.agent-loop/fix-prompt.md` is regenerated from the Claude-owned review issues, and the cycle repeats from step 3 within `max_cycles`.
9. The cycle stops for human approval before any commit, and again before the next phase begins.

`.agent-loop/loop-state.json` is an orchestrator-owned runtime artifact in the shipped system: the activator initializes it at phase activation and `scripts/agent_loop.py` updates the per-cycle fields (`status`, `cycle_count`, `last_verdict`, `last_verdict_phase`, `claude_version`, `codex_version`, `orchestrator_version`, `approval_mode`, `awaiting_human_for`) during a run, while the Codex- or human-owned fields (`phase`, `sub_phase`, `task`, `max_cycles`, `contract_version`) come from the Phase 4C activation step. In the manual-fallback flow above a human edits the same orchestrator-writable fields against the Phase 3A Orchestrator Contract write-ownership rules; the file is never recorded by hand as the default workflow.

The operator flow across Phase 4B + Phase 4C is:

1. Run `python scripts/agent_loop.py plan` to generate `.agent-loop/proposed-phase.md`. The planner enforces every Phase 4A refusal condition (unresolved `NEEDS_FIXES`, `FAILED_REQUIRES_HUMAN`, any `halted_*` status, orchestrator-in-flight statuses, `cycle_count >= max_cycles` on `NEEDS_FIXES`, stale evidence, unreadable evidence headers, prior approved proposal carrying `APPROVED_FOR_ACTIVATION`, malformed `loop-state.json`) and the bounded-scope rules (max 10 files, max 1 contract revision, testable Definition-of-done bullet, no vague Objective language, cycle-size estimate <= 3 without justification, no `active-in-flight` / `pending` dependency, no label collision against historical sub-phases, no `APPROVED_FOR_ACTIVATION` token authored by the planner itself, `## Exclusions` enumerating later 4x / 5+ sub-phases). The planner writes only `.agent-loop/proposed-phase.md` and a `note:`-style line in `.agent-loop/planner.log`.
2. A human reviews the proposal and, if accepted, appends a `## Approval` section to `.agent-loop/proposed-phase.md`. The approval section must contain the literal `APPROVED_FOR_ACTIVATION` token on its own line within the section AND must reference the proposal's `## Label` text. Token forgery (extra words on the line, alternate capitalization, surrounding characters) and stale-label approvals are refused at activation time.
3. Run `python scripts/agent_loop.py activate` to consume the approved proposal. The activator parses the proposal, verifies the approval requirements exactly, refuses cleanly on every failure mode (missing `## Approval` section, malformed token, label mismatch, missing/unreadable proposal, missing/malformed `loop-state.json`, empty required-proposal-section bodies), and on success performs the activation write set plus the Phase 5F phase-start prompt bootstrap in one fail-closed path: rewriting `TASK.md`'s `## Active Phase` / `## Active Sub-Phase` / `## Phase Status` / `## Active Task` / `## Phase Outcome Required Now` / `## Next-Phase Gate` / `## Out Of Scope For Current Phase` (preserving `## Human Objective` and `## Project Intent` verbatim); rewriting `.agent-loop/current-task.md` and `.agent-loop/current-phase.md`; updating `.agent-loop/phase-plan.md`'s `## Active Phase` line, closing the previously-active sub-phase's `### Status` opening line as a transition note, and APPENDING a new sub-phase section that mirrors the activated proposal; resetting `.agent-loop/loop-state.json` to `status = awaiting_claude_implementation`, `cycle_count = 0`, `last_verdict = null`, `last_verdict_phase = null` with `phase` / `sub_phase` / `task` set from the approved proposal and `max_cycles` / `contract_version` / `claude_version` / `codex_version` / `orchestrator_version` preserved per the Phase 3A write-ownership rules; writing `.agent-loop/claude-prompt.md` inside the same atomic-or-rollback activation transaction; appending a `note:`-style approval-source line (file path, mtime, literal approval line) to `.agent-loop/planner.log`; and appending a `prompt bootstrap:` audit note to `.agent-loop/orchestrator.log`. The activator exits 0 on success and 2 on any refusal; refusals never partially write any activation file.

Phase 4D integration (shipped): after the orchestrator persists a terminal `APPROVED_FOR_HUMAN_REVIEW` verdict in `_handle_verdict_loop`, it invokes the standalone planner exactly once to refresh `.agent-loop/proposed-phase.md`, then logs a single `note:`-style line to `.agent-loop/orchestrator.log` recording the planner's exit code. The orchestrator still returns 0 after the terminal approval in all of those cases. The hook never auto-activates a proposal and never widens the planner's write boundary (only `.agent-loop/proposed-phase.md` and `.agent-loop/planner.log` may be written by the planner call); approval and activation remain the separate, human-controlled `activate` step described above.

Phase 4E adds a planner-adapter seam. Both the `plan` path (step 1) and the post-approval planner refresh (step 4) now dispatch planner execution through `make_planner_adapter().run(...)` instead of calling the planner in-process directly. The default adapter (`LocalPlannerAdapter`) runs the same in-process planner, so behavior, refusal handling, the exception-containment guarantees on the post-approval path, and the planner's write boundary (`.agent-loop/proposed-phase.md` and `.agent-loop/planner.log` only) are all unchanged; the seam exists so a later slice can supply an alternate planner adapter without touching the call sites. The adapter is purely a dispatch boundary: it never widens the planner write boundary and never activates a proposal. Activation stays the separate, human-controlled `activate` step - planning (proposal generation and refresh) and activation remain distinct.

Phase 5B (Review Mode Initial Slice, shipped) makes `review` the explicit default approval mode and adds the first machine-readable Claude completion handoff artifact. New / freshly activated Phase 5+ runtime state in `.agent-loop/loop-state.json` carries `approval_mode = "review"` and `awaiting_human_for = null` by default (a pre-existing non-review `approval_mode` is preserved across activation; `awaiting_human_for` is always reset by activation). During a normal cycle the orchestrator backfills the same defaults on any loaded state that pre-dates Phase 5B, and on the implemented `review` path it sets `awaiting_human_for = "phase_complete_awaiting_human_approval"` on `APPROVED_FOR_HUMAN_REVIEW` and leaves it `null` everywhere else. After Claude returns a fresh `claude-summary.md` for either an implementation or fix cycle and evidence succeeds, the orchestrator writes `.agent-loop/claude-done.json` (`signal_version`, `phase`, `sub_phase`, `task`, `cycle_count`, `mode` of `"implementation"` or `"fix"`, `source_prompt_path`, and `status = "ready_for_codex_review"`) as a routing/timing signal only; it never substitutes for `claude-summary.md`, the git diff, or validation evidence in review. Each new prompt or fix-prompt issuance clears any prior `claude-done.json` so a stale completion signal cannot drive the wrong review cycle. The shipped `review`-mode baseline is preserved unchanged by Phase 5C.

Phase 5C (Strict Mode Pauses, shipped) implements the three `strict`-mode human checkpoints defined in the Phase 5A contract. When `approval_mode = "strict"`, the orchestrator pauses at: (1) before a new implementation prompt dispatches to Claude (gate name `pre_claude_prompt`, halt status `halted_awaiting_human_pre_claude_prompt`); (2) before a new fix prompt dispatches to Claude (`pre_fix_prompt`, `halted_awaiting_human_pre_fix_prompt`); (3) after Claude completion and evidence validation but before Codex review begins (`pre_codex_review`, `halted_awaiting_human_pre_codex_review_normal` for the implementation cycle and `halted_awaiting_human_pre_codex_review_fix` for the fix cycle - two halt-status flavors so resume can route to the right continuation). At each gate the orchestrator persists `awaiting_human_for = <gate name>` plus the matching halt status in a single atomic write, logs a `note:` line to `.agent-loop/orchestrator.log`, and exits cleanly with code 2. After the human approves the paused step they run `python scripts/agent_loop.py resume`, which refuses unless `status` is one of the four strict-gate halts, then clears `awaiting_human_for`, restores a ready-to-continue status, and dispatches to the continuation matched by the persisted gate halt-status so no earlier work is re-done and the next gate (if any) still fires.

Phase 5D (Autonomous Mode Initial Slice, shipped) implements the first narrow `autonomous` runtime continuation path. When `approval_mode = "autonomous"`, the orchestrator bypasses each of the three Phase 5C strict-mode human gates (`pre_claude_prompt`, `pre_fix_prompt`, and `pre_codex_review` - which fires once per cycle and persists in two halt-status flavors `halted_awaiting_human_pre_codex_review_normal` / `halted_awaiting_human_pre_codex_review_fix` so resume can route to the matching implementation-cycle or fix-cycle continuation) and continues directly through the existing implementation step, the existing Claude-completion-to-Codex-review handoff, and bounded `NEEDS_FIXES` fix cycles within the existing `cycle_count` / `max_cycles` threshold. Each gate bypass writes a single `autonomous mode: bypassing <gate> gate at <where>` note to `.agent-loop/orchestrator.log` so the mode is auditable from repo artifacts and not hidden in transient control flow. Every existing hard stop is preserved: `FAILED_REQUIRES_HUMAN` still halts immediately, malformed evidence or summaries still fail closed, the `cycle_count >= max_cycles` threshold on `NEEDS_FIXES` still halts, and the `APPROVED_FOR_HUMAN_REVIEW` terminal still persists `status = phase_complete_awaiting_human_approval` and `awaiting_human_for = phase_complete_awaiting_human_approval` because the Phase 5A contract requires human approval before phase progression. The shipped `review` (Phase 5B) and `strict` (Phase 5C) behavior is unchanged.

Phase 5E (Post-Review Artifact Reconciliation, shipped) makes the post-review follow-up step deterministic and ownership-aware. Every review issue is classified Codex-owned vs Claude-owned: an explicit `Owner:` field is honored if present, otherwise inference is based on the issue's `File(s)` partition. Inference returns Codex when every listed file is in `CODEX_OWNED_REVIEW_PATHS`, Claude when every listed file is outside that set, and refuses at parse time (`halted_review_parse_failed`) when the file list mixes Codex-owned planning paths with Claude-owned implementation paths (the contract's "refuse when ownership is ambiguous" rule). Each supported Codex auto-fix action declares a per-action allow-list of paths it may touch (`CODEX_ACTION_ALLOWED_FILES`); a Codex-owned issue whose `File(s)` list extends outside its action's allow-list is refused at reconciliation time (`halted_input_missing`) before any side-effect, which is the contract guarantee that a Codex auto-fix cannot overwrite Claude-owned implementation work. Reconciliation writes an up-front `review reconciliation: codex_owned_issues=N (actions=[...]) claude_owned_issues=N` summary to `.agent-loop/orchestrator.log`, keeps the per-action and fix-prompt-synchronization notes from earlier phases, and now reloads loop-state from disk on the reconciliation halt path so a successful Codex auto-fix (e.g. `sync_phase5_runtime_defaults`) persists across the subsequent halt instead of being clobbered by the pre-reconciliation snapshot. The shipped `review` (Phase 5B), `strict` (Phase 5C), and `autonomous` (Phase 5D) runtime behavior is unchanged; Claude-owned findings still drive `.agent-loop/fix-prompt.md` regeneration in the required handoff format.

Phase 6G (Automatic Continuation Chaining Initial Slice, shipped) builds on the shipped 6D / 6E / 6F primitives by adding bounded automatic continuation chaining over the existing single-hop token-exhaustion resume. The operator runtime entry is the new `auto-continue` subcommand (`python scripts/agent_loop.py auto-continue`): while `.agent-loop/loop-state.json` is at `halted_awaiting_token_exhaustion_continuation`, the chain repeatedly applies the shipped Phase 6F `run_token_exhaustion_resume` hop until one of these stop conditions: (a) a hop's return code is non-zero (a refusal - budget exhausted, malformed checkpoint, unsupported saved stage, cycle-identity mismatch - or a non-token halt inside the dispatched continuation), in which case the chain propagates the rc unchanged; (b) the cycle has cleared the token-exhaustion halt after a successful hop (no fresh exhaustion event was recorded during the dispatched continuation), in which case the chain returns 0; or (c) `AUTO_CONTINUE_MAX_HOPS = 4` hops have run, in which case the chain refuses fail-closed as a defense-in-depth bound independent of the per-checkpoint `continuation_budget`. Each chain hop logs an `auto-continue chain hop N begin` audit line so the chain is reconstructable from `.agent-loop/orchestrator.log` after the fact. After every successful hop the chain re-runs the same `validate_loop_state` and `check_contract_version` structural / contract checks the chain entry used BEFORE branching on the persisted halt status, so a dispatched continuation that left loadable-but-invalid loop-state on disk (missing required field, unsupported `contract_version`, etc.) is refused fail-closed via `_halt` rather than logged as a chain completion or silently advanced into another hop. The initial "status is not the token-exhaustion halt" check uses the recovery-preserving refusal pattern (NOT `_halt`), so running `auto-continue` against a strict-gate halt by mistake does not clobber the strict-gate recovery point. The chain is OPT-IN: the existing `resume` subcommand still dispatches the single-hop resume on a `HALTED_TOKEN_EXHAUSTION` halt; only the dedicated `auto-continue` subcommand enters the chain. Phase-boundary memory distillation, repeated-failure memory synthesis, and broader optional-context file loading remain deferred. Every shipped Phase 5 behavior (`review`, `strict`, `autonomous`, post-review reconciliation, and the phase-start prompt baseline) remains unchanged.

Phase 6H (Bounded Continuation Prompt Construction Initial Slice, shipped) builds on the shipped 6B (storage), 6C (advisory-only retrieval), 6D (checkpoint storage), 6E (checkpoint resume), 6F (token-exhaustion continuation), and 6G (auto-continuation chaining) primitives. The new `build_continuation_prompt_context(repo_root, *, memory_entry_limit=None, evidence_byte_limit=None, log_path=None) -> dict` library function assembles a structured continuation-context payload from canonical loop-state + the active Phase 6F/6G token-exhaustion checkpoint + bounded evidence excerpts + advisory durable-memory entries. The dict carries a top-level `context_signal_version = "phase-6h-v1"`, a `canonical_state` subset of loop-state (phase / sub_phase / task / cycle_count / approval_mode / halt_status / awaiting_human_for), a `checkpoint` subset of the active checkpoint (suspension_reason, active_prompt_path, continuation_budget, pre_suspension_status, pre_suspension_awaiting_human_for, source_path, source_artifact_path, created_at), an `evidence` map of each `EVIDENCE_FILES` rel path -> bounded excerpt metadata (`{absent: True}` for a missing file, otherwise the full readable-file block `{absent: False, byte_size_on_disk, excerpt, excerpt_byte_size, truncated}`; an evidence file that exists but cannot be read refuses fail-closed rather than emitting an undocumented `read_error` branch), a `memory_advisory` list of Phase 6C-retrieved entries (each carrying `advisory_only = True`; the `checkpoint` memory category is intentionally excluded because the active checkpoint is already on the dedicated `checkpoint` key), `memory_limit_applied` / `memory_truncated_at_limit` flags, `evidence_byte_limit_applied`, and an explicit `canonical_precedence_note` reminder that canonical artifacts win over advisory memory. The operator runtime entry is the new `build-continuation-context` subcommand which persists the JSON payload to `.agent-loop/continuation-context.json` (override with `--output`) so the construction is auditable from on-disk artifacts; it accepts `--memory-entry-limit` (default `CONTINUATION_CONTEXT_DEFAULT_MEMORY_LIMIT = 8`, capped at `MEMORY_RETRIEVAL_MAX_LIMIT = 32`) and `--evidence-byte-limit` (default `CONTINUATION_CONTEXT_DEFAULT_EVIDENCE_BYTE_LIMIT = 4096` per file, capped at `CONTINUATION_CONTEXT_MAX_EVIDENCE_BYTE_LIMIT = 65536`). Refusal modes (all fail-closed): missing / malformed `loop-state.json`, unsupported `contract_version`, halt status outside the eligible set (only `HALTED_TOKEN_EXHAUSTION` in this initial slice; strict-gate halts continue using their existing `resume` path), no active checkpoint on disk, cycle-identity mismatch between checkpoint and loop-state, `suspension_reason` other than `token_exhaustion`, out-of-bound memory or evidence limits (zero, negative, non-int, bool, above the cap), and an evidence file that exists on disk but cannot be read. The library function never writes to canonical artifacts and is byte-equivalent on loop-state and the checkpoint across a build call. Prompt string serialization, phase-boundary memory distillation, repeated-failure synthesis, and broader optional context-file loading beyond the existing `EVIDENCE_FILES` set remain deferred. Every shipped Phase 5 behavior remains unchanged.

Phase 6I (Phase-Boundary Memory Distillation Initial Slice, shipped) builds on the shipped 6B (storage), 6C (retrieval), 6D (checkpoint storage), 6E (resume), 6F (token-exhaustion continuation), 6G (auto-continue), and 6H (continuation context) primitives. The new `distill_phase_boundary_memory(repo_root, *, excerpt_byte_limit=None, log_path=None) -> list[Path]` library function writes append-mostly durable memory entries in three of the five Phase 6A categories - `summary`, `decision`, and conditionally `failure` - at a successfully APPROVED phase boundary, sourced from canonical loop-state plus bounded excerpts of the canonical source artifacts `.agent-loop/claude-summary.md` and `.agent-loop/codex-review.md`. Each entry body carries an explicit `distillation_signal_version = "phase-6i-v1"` marker plus a `source_artifacts` list pointing at the canonical sources so a later retrieval can trace provenance. Phase 9F (Capacity-Halt Reprobe And Automatic Resume, complete) ships the bounded capacity-halt re-probe + automatic-resume layer for Claude/Codex token-quota or rate-limit exhaustion. The shipped runtime treats `halted_capacity_unavailable` as a resumable EXTERNAL-CAPACITY halt distinct from the Phase 6F token-exhaustion continuation seam (which writes a new continuation prompt and resumes within the same cycle); Phase 9F waits for the external capacity to return, re-probes through a configurable probe callable, then restores the pre-suspension loop-state `status` so the orchestrator resumes the exact suspended step the next time it runs. The library function `reprobe_capacity_and_resume(repo_root, *, max_attempts=None, initial_backoff_seconds=None, max_backoff_seconds=None, suspended_status=None, probe=None, sleep=None, output_path=None, log_path=None) -> Path` reads validated loop-state, requires `status == halted_capacity_unavailable`, loads or initializes the retry-state JSON at `.agent-loop/capacity-retry-state.json`, and runs up to `max_attempts` bounded retry attempts. Each attempt computes `backoff = min(initial_backoff_seconds * 2 ** (attempt_index - 1), max_backoff_seconds)` so the wait is exponentially bounded, invokes `sleep(backoff)` (default `time.sleep`; injected callable for tests), then invokes `probe(repo_root) -> bool`. The default probe is env-var-gated through `AGENT_LOOP_CAPACITY_PROBE`: it returns True only when the env var is set to a recognized truthy value (`available`, `1`, `true`, `yes`, `ok` - case-insensitive); any other value (or an unset env var) returns False so the bounded retry budget exhausts rather than silently auto-succeeding without a real availability check. Operators with a real capacity check inject a custom `probe` callable via the library entry point. The Phase 9F seam stops short of parsing the review or driving the verdict loop; `_handle_verdict_loop` remains the canonical owner of post-review routing, which the next orchestrator step (Phase 9E review/fix iteration via `run_internal_review_fix_cycle(...)` or the shipped Codex review continuation a normal cycle drives) consumes.
